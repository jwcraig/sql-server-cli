# sscli SQL Command And Admin Workflows Spec

Status: proposed
Author: Codex
Created: 2026-03-22
Updated: 2026-03-22

## Summary

This spec proposes reshaping `sscli` so it becomes the preferred raw SQL tool
for agent workflows that currently fall back to `sqlcmd`.

The core recommendation is:

- keep `sql` as the canonical raw SQL command
- add a top-level SQL shorthand:
  - `sscli "SELECT * FROM dbo.inventory"`
- expand `sql` to support operational and admin workflows without forcing users
  into a separate `exec` command
- preserve dedicated discovery and diagnostics commands such as `describe`,
  `tables`, `status`, and future DBA helpers
- keep the existing connection/profile model
- remove implicit `.env` auto-loading from the working directory

This spec also proposes first-class support for:

- robust `GO` batch execution
- SQL Server session `SET` options and presets
- execution plans and runtime diagnostics
- built-in DBA helpers such as `table-size`, `index-usage`, and `log-usage`

## Problem

The main problem is not that `sscli` lacks a better command name. The main
problem is that agents and operators still reach for raw `sqlcmd` when they
need full SQL Server behavior.

Common fallback cases:

- multi-statement DDL scripts
- scripts that require `GO` batch separators
- filtered indexes and other DDL that require specific session settings
- `DBCC` operations such as `DBCC SHRINKFILE`
- plan analysis and runtime diagnostics
- direct DMV-based storage and index analysis
- long SQL scripts from files or stdin

The current `sql` command is documented as read-only by default. That framing
creates friction for the very users who want `sscli` to replace `sqlcmd` for
real operational work.

The current implicit `.env` auto-loading behavior also creates confusion for
agents because connection selection can appear to change based on ambient files
in the working tree.

## Goals

- Make raw SQL execution in `sscli` feel as direct as `sqlcmd`.
- Reduce the need to fall back to `sqlcmd`.
- Preserve deterministic output for agents and automation.
- Preserve exact or near-exact SQL Server semantics for batch execution.
- Keep existing profile/config connection workflows.
- Remove hidden connection surprises from implicit `.env` discovery.

## Non-Goals

- Renaming the main raw SQL surface away from `sql`.
- Introducing a new top-level connection model such as `--target`.
- Replacing every SQL Server feature with a bespoke helper command.
- Hiding SQL Server behavior behind opaque abstractions.
- Removing the ability to run arbitrary SQL.

## Proposed Command Model

### Canonical raw SQL command

Use `sql` as the canonical command for arbitrary SQL execution:

```bash
sscli sql "SELECT TOP 10 * FROM dbo.T_MP"
sscli sql --json "SELECT name FROM sys.tables ORDER BY name"
sscli sql --file db/migrations/0005-optimize-v2-indexes.sql
cat patch.sql | sscli sql --stdin
```

Rules:

- `sql` is the documented raw SQL command.
- `sql` supports read-only, write, DDL, admin, and batch execution.
- `sql` is the default path for ad hoc SQL typed by humans or generated by
  agents.
- `sql` remains compatible with the current JSON contract where practical.
- `sql` no longer relies on parser-enforced read-only validation in the shipped
  distribution.

### Bare SQL shorthand

Add a top-level shorthand so quoted SQL can be passed without the `sql`
subcommand:

```bash
sscli "SELECT * FROM dbo.inventory"
sscli --json "SELECT COUNT(*) FROM sys.tables"
```

Behavior:

- if the first non-flag token does not match a known subcommand and looks like
  SQL, treat it as `sql <text>`
- known subcommands such as `status`, `tables`, `describe`, and `compare` keep
  their current meaning
- help text and shell completions should continue to center `sql` as the
  canonical form; the bare form is shorthand, not a new separate command family
- shorthand mode is intentionally narrow:
  - it is for inline SQL text only
  - global flags such as `--json`, `--markdown`, `--pretty`, connection flags,
    and `--env-file` may appear before the SQL text
  - SQL-specific flags such as `--file`, `--stdin`, `--param`, `--dry-run`,
    `--continue-on-error`, and future `--set` or `--preset` options require the
    explicit `sql` subcommand

Recommended detection rules:

- treat input as SQL when the first non-flag token:
  - contains whitespace, or
  - begins with a common SQL keyword such as `SELECT`, `WITH`, `INSERT`,
    `UPDATE`, `DELETE`, `MERGE`, `CREATE`, `ALTER`, `DROP`, `EXEC`, `DBCC`,
    `DECLARE`, or `USE`
- do not silently reinterpret a misspelled known subcommand as SQL unless it
  satisfies the SQL detection rules above

### Optional alias

`query` may remain as a visible alias to `sql` for discoverability, but it is
not the primary concept in this spec.

If retained:

- `query` should be a strict alias of `sql`
- it should not carry different safety or capability semantics
- docs should continue to teach `sql`

### Introspection and diagnostics

Keep and expand dedicated commands:

- `describe`
- `tables`
- `status`
- `plan`
- `profile`
- `io`
- `table-size`
- `index-usage`
- `missing-indexes`
- `files`
- `log-usage`
- `views`

These commands remain important because agents should not have to hand-write
catalog or DMV SQL for common inspection workflows.

## Safety Philosophy

This proposal favors capability and explicit behavior over parser-enforced
friction.

Recommended model:

- shipped `sscli` should support full SQL execution
- `sql` should not be positioned as a read-only-only command
- lightweight safety rails are acceptable
- hidden parser restrictions that push users back to `sqlcmd` are not
- `--allow-write` should be deprecated and then removed
- `allowWriteDefault` should be deprecated and then removed

Migration guidance:

- during the transition period, `--allow-write` may remain accepted as a no-op
  compatibility flag
- `allowWriteDefault` may remain accepted in config for one or more releases but
  should no longer affect execution semantics
- README and help text should stop teaching read-only-by-default behavior once
  this transition begins

Recommended runtime behavior:

- always print resolved server and database before executing SQL unless
  suppressed
- return non-zero on execution failure
- keep deterministic JSON output for automation
- optionally support a TTY confirmation path for obviously destructive
  operations, gated by a flag such as `--yes` or `--force`

Out of scope for this spec:

- source-build variants that remove write capability
- local wrappers or custom forks that hard-disable mutations

Those remain valid deployment choices, but they should not define the primary
product behavior.

### Output stream rules for safety banners

To preserve JSON automation behavior:

- target and safety banners must go to stderr, not stdout
- `--json` mode must still emit exactly one JSON object on stdout
- `--quiet-target` suppresses the stderr target banner

## Connection And Environment Model

Do not change the existing connection model.

Continue to use:

- profiles
- config files
- explicit CLI flags such as `--server`, `--database`, and `--user`
- explicit environment variables

Do not add:

- a new `--target` abstraction
- new connection string shorthand syntax as part of this proposal

### `.env` behavior

Remove implicit `.env` auto-loading from the current working directory.

Recommended behavior:

- environment variables in the parent shell remain supported
- `--env-file path` remains supported
- if users want `.env` semantics, they must opt in with `--env-file`
- if `--env-file` points to a missing or unreadable file, return an error

Rationale:

- implicit `.env` discovery is convenient for local development
- it is confusing for agent workflows because connection selection becomes
  ambient rather than explicit
- explicit `--env-file` preserves the convenience without the hidden behavior

## SQL Input And Batch Execution Requirements

`sql` must support real SQL Server script workflows:

- inline SQL text
- `--file path.sql`
- `--stdin`
- multi-statement scripts
- `GO` batch separators
- sequential batch execution with clear error reporting

Examples:

```bash
sscli sql --file db/migrations/0005-optimize-v2-indexes.sql
cat patch.sql | sscli sql --stdin
sscli "SELECT 1"
```

Expected behavior:

- preserve batch boundaries faithfully
- show which batch failed
- return non-zero on failure
- optionally continue across batches with `--continue-on-error`
- surface batch timing and row counts in JSON output

### `GO` semantics

`GO` handling must be robust enough for real migration scripts.

Minimum required behavior:

- treat `GO` as a batch separator only when it appears alone on a logical line,
  ignoring surrounding whitespace
- match `GO` case-insensitively
- support `GO n` repeat counts
- do not treat `GO` inside string literals as a separator
- do not treat `GO` inside line or block comments as a separator
- tolerate leading UTF-8 BOM in files
- ignore empty batches created by consecutive separators unless a repeat count
  makes them meaningful
- preserve the original SQL text of each batch for error reporting and dry-run
  output

The spec does not require reproducing every obscure `sqlcmd` edge case on day
one, but it does require behavior that is reliable for normal SQL Server
migration and admin scripts.

## Session Option Support

Some SQL Server features require specific session settings. This must be
first-class in `sscli`, not a hidden implementation detail.

Support:

```bash
sscli sql \
  --set ANSI_NULLS=ON \
  --set ANSI_PADDING=ON \
  --set ANSI_WARNINGS=ON \
  --set ARITHABORT=ON \
  --set CONCAT_NULL_YIELDS_NULL=ON \
  --set QUOTED_IDENTIFIER=ON \
  --set NUMERIC_ROUNDABORT=OFF \
  --file create-index.sql
```

Add presets:

- `--preset ddl-safe`
  - applies the SQL Server settings required for filtered indexes, indexed
    views, computed-column indexes, and similar DDL
- `--preset plan-safe`
  - stable session settings for plan and profiling workflows

Required semantics:

- presets expand to a documented list of `SET` statements
- explicit `--set` values override conflicting values from presets
- presets and `--set` options establish the initial session state before the
  first user batch is executed
- user SQL may still change session settings later in the script, and `sscli`
  must not reapply presets before every batch unless an explicit future mode is
  added for that behavior
- dry-run output shows the effective settings that will be applied

Example:

```bash
sscli sql --preset ddl-safe --file db/migrations/0005-optimize-v2-indexes.sql
```

## Diagnostics And Plan Tools

Add first-class plan and runtime diagnostics:

```bash
sscli plan "SELECT Nr FROM dbo.T_MP WHERE Nr LIKE '001-01%'"
sscli profile "SELECT Nr, MP FROM dbo.T_MP WHERE E7 IS NULL ORDER BY E2, E3, E4"
sscli io "SELECT COUNT(*) FROM dbo.measurement WHERE data_set_id = 42"
```

### `plan`

Purpose:

- estimated access path
- object/index selection
- predicate visibility

Expected output:

- operator tree
- object name
- index name
- seek predicate
- residual predicate
- estimated rows

### `profile`

Purpose:

- execution profile with operator row counts

Expected output:

- physical operator tree
- estimated vs actual rows where available
- high-cost operators highlighted

### `io`

Purpose:

- `STATISTICS IO`
- `STATISTICS TIME`

Expected output:

- table reads
- logical reads
- elapsed time
- CPU time

## Built-in DBA Helpers

These commands should wrap common DMV workflows that operators and agents
currently hand-write repeatedly.

### `db-size`

Show:

- database total size
- data file size
- log file size
- used vs free space

### `table-size`

Show:

- row counts
- data MB
- index MB
- total MB

### `index-size`

Show:

- per-index size for a given table

### `index-usage`

Show:

- seeks
- scans
- lookups
- updates
- server restart time

### `missing-indexes`

Show:

- missing-index DMV suggestions
- estimated impact
- equality columns
- inequality columns
- included columns

### `files`

Show:

- database files
- logical name
- physical path
- allocated MB
- used MB
- free MB

### `log-usage`

Show:

- log size
- used MB
- used percent

### `views`

Show:

- schema
- view name
- schema-bound status
- indexed-view status

## Output Requirements

Every relevant command should support the output modes needed by both humans
and automation.

For `sql`, `plan`, `profile`, `io`, and relevant helper commands, support:

- `--json`
- `--raw`
- `--no-truncate`
- `--max-rows N`
- `--delimiter '|'`
- `--quiet-target`
- `--no-warn`

Recommended defaults:

- human mode: compact markdown
- automation mode: `--json`
- plan/diagnostic mode: `--raw --no-truncate`

### JSON stability

The existing `sql` JSON contract should be preserved where practical:

```json
{ "success": true, "batches": [...], "resultSets": [...], "csvPaths": [...] }
```

Requirements:

- do not rename the top-level `sql` JSON shape merely because functionality is
  expanded
- extend fields additively where possible
- document new JSON shapes for `plan`, `profile`, `io`, and new helper commands
- keep stdout to a single JSON object in `--json` mode
- any human-readable execution banners go to stderr

## Implementation Plan

### Phase 1: Simplify the SQL surface

- keep `sql` as the canonical raw SQL command
- add top-level bare-SQL shorthand
- decide whether `query` remains as a strict alias or is removed from user docs
- update help text and README to teach `sql` first
- deprecate `--allow-write` and `allowWriteDefault`

### Phase 2: Remove ambient config surprises

- remove implicit `.env` auto-loading
- keep explicit `--env-file`
- document connection precedence clearly without changing profile behavior

### Phase 3: Batch and session support

- add `--stdin`
- implement robust `GO` support
- add `--set` options
- add `--preset ddl-safe`
- add `--preset plan-safe`

### Phase 4: DBA helpers

- add `files`
- add `log-usage`
- add `table-size`
- add `index-usage`
- add `missing-indexes`

### Phase 5: Plan and diagnostics

- add `plan`
- add `profile`
- add `io`
- add stable raw and JSON output shapes

## Acceptance Criteria

- `sql` is the documented and canonical raw SQL command.
- `sscli "SELECT 1"` works as shorthand for raw SQL execution.
- bare-SQL shorthand remains limited to inline SQL plus global flags.
- `sql --file` and `sql --stdin` support multi-batch scripts.
- `GO` handling is robust enough for real migration and admin scripts.
- `GO n` repeat counts are supported.
- filtered-index creation succeeds with `--preset ddl-safe`.
- users can inspect file usage, log usage, table size, and index usage without
  hand-writing DMV SQL.
- users can inspect estimated plans and runtime profiles without falling back to
  `sqlcmd`.
- the tool no longer implicitly auto-loads `.env` from the working directory.
- `--env-file missing.env` returns an error.
- the existing connection/profile model remains intact.
- `--json` mode still emits a single JSON object on stdout, with target banners
  on stderr.

## Open Questions

- Should `query` remain as a visible alias to `sql`, or should docs stop
  mentioning it entirely?
- Should `sql` support optional TTY confirmation for obviously destructive
  statements, or should it remain non-interactive by default?
- Should there be a `shrink-log` helper command, or should `sql` plus
  `log-usage` be considered sufficient?

## Future Work: Safe Custom Builds

Some users may want a locked-down distribution that strips write capability or
removes certain commands entirely.

That should not change the default shipped behavior of `sscli`, but it may be a
useful future capability.

Recommended position for now:

- shipped `sscli` is a full-capability tool
- organizations that need a completely safe tool may maintain a custom build or
  wrapper
- a formal capability manifest is a separate design problem, not part of this
  core spec

If pursued later, a separate spec should define:

- capability keys such as raw-sql-write, admin-diagnostics, or command-level
  enablement
- compile-time or packaging-time command registration rules
- help and completion behavior when commands are removed
- runtime behavior for disabled functionality
- tests proving stripped functionality is absent rather than merely undocumented

## Recommended Immediate Next Steps

1. Update the spec and README direction so `sql` remains canonical.
2. Add top-level bare-SQL shorthand.
3. Remove implicit `.env` auto-loading, require explicit `--env-file`, and
   error on missing env files.
4. Deprecate `--allow-write` and `allowWriteDefault`.
5. Implement robust `GO` batch support under `sql`.
6. Add `--preset ddl-safe`, then `files`, `log-usage`, `table-size`, and
   `index-usage`.
